# 领域逻辑与数据访问

## `assessment.ts`

该文件是测评系统的唯一业务规则来源，包含：

- 题库、题目维度和选项计分
- 10 / 15 / 20 题三轮配置
- 第二轮的弱信号校准和第三轮的强信号深挖
- 客户端与服务端共用的累计题序验证
- MBTI、Big Five、价值、关系、调节、弹性、边界与双向伴侣吸引画像生成
- 双层答案计分：每题一个主选，可选一个次选；次选以 0.45 权重参与评分与自适应选题
- 第二、三轮关系情境题与四组伴侣偏好、四组对外吸引信号；性别资料不进入任何计分公式

### 题库写作规范

当前完整候选题库共 66 题，每次探索按 10 / 15 / 20 题渐进呈现，共实际作答 45 题。新增或修改题目时必须遵守：

- 题干只问一个判断，并把必要背景放在场景区；题干不超过 28 个字符，场景不超过 72 个字符。
- 行为题优先限定到“最近三个月”“第一反应”“前十分钟”或明确截止时间，避免询问理想中的自我。
- 价值题必须设置相同条件或明确的取舍，例如“只能保留一种”，避免四个答案都正确。
- 每题固定四个选项，动作、时间点或核心动机应能彼此区分；幽默放在场景和说明中，不能遮住选项含义。
- 每个 `focus` 维度必须被至少一个选项实际计分，否则会让自适应选题追踪一个没有证据的信号。
- 主选记录最常发生或最先发生的反应；只有真正并列时才使用次选，不能用次选掩盖含糊题目。

自适应选题必须是确定性的：同一组历史答案应得到同一题序，否则服务端无法验证客户端提交。修改题目 ID、排序或计分后，务必运行 `npm test`。

## `profile.ts`

人物资料统一使用 `ParticipantProfile`：昵称加 `male / female / nonbinary / undisclosed` 四种性别值。新的本地存储键为 v2；旧昵称只用于预填迁移，必须由用户补选性别后才形成有效人物卡。性别可以写入 v3 报告快照用于展示，但不得用于推断性取向、伴侣偏好或管理员统计。

## `data.ts`

该文件负责用户和报告的数据访问。所有公开函数都会先确保 D1 基础结构存在，以兼容已部署的旧实例；正式模式变更仍必须生成 Drizzle 迁移。

管理员规则：若配置 `ADMIN_EMAILS`，匹配用户会在下次访问时提升为管理员；未配置且数据库为空时，首位用户作为初始管理员。
